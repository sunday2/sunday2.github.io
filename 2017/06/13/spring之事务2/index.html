spring之事务(2)

发表于 2017-06-13 | 分类于 java |

其实第一篇讲的偏理论，比较抽象，现在说说在spring框架中如何使用事务。

基本配置

使用事务的前提当然是使用了是数据库了，不管是mysql还是oracle,引入数据库相关的jar是必须的，比如数据库驱动jar,数据库连接池jar，和spring整合的jar，spring本身的jdbc jar，spring-jdbc-xxx.jar都是必须的。

除此之外，因外我们主要讲事务所以我们还要引入spring的事务的jar包，spring-tx-xxx.jar。除此之外，如果想要更灵活的配置事务，可能还需要引入spirng-aop-xxx.jar。

另外要注意一点，有些人会说maven配置依赖的话，有些包本身就依赖了其它jar包，可能就不需要另外指定了。确实如此，但是我们要注意它们的范围scope。

举个例子：使用mybatis数据框架的话，我们需要添加spring对mybatis的整合包mybatis-spring-xxx.jar是毋庸质疑的,这个jar包本身也是用maven构建依赖，我们可以看到这个jar包本身就依赖了一些其它spring框架的jar包。我们可以找到这个jar并查看它的pom文件。

provided和test在打包的时候都不会将依赖打包(所以有时候发现编译和测试时没问题，部署到环境中就少包)，所以我们还是要在自己项目的pom文件指定相关的依赖。

spring中的声明式事务

一般是在service层使用事务。

官方指导手册有一段话没太懂：

If the DataSource, used by any non-JTA transaction manager, is looked up via JNDI and
managed by a Java EE container, then it should be non-transactional because the Spring
Framework, rather than the Java EE container, will manage the transactions.

<!--定义一个jdbc数据源-->
<bean id="dataSource" class="org.apache.commons.dbcp.BasicDataSource" destroy-method="close">
<property name="driverClassName" value="${jdbc.driverClassName}" />
<property name="url" value="${jdbc.url}" />
<property name="username" value="${jdbc.username}" />
<property name="password" value="${jdbc.password}" />
</bean>

<!--如果在jee容器中使用jpa，则通过jndi配置数据源-->
<jee:jndi-lookup id="dataSource" jndi-name="jdbc/jpetstore"/>


<!--定义jdbc事务管理器，并引用一个数据源。注意spring中的事务管理器有两种，一种是jdbc一种是jta，区别在于单数据库和多数据库区别-->
<bean id="txManager" class="org.springframework.jdbc.datasource.DataSourceTransactionManager">
<property name="dataSource" ref="dataSource"/>
</bean>

<!--如果是jta事务管理器，这里没有引用数据源-->
<bean id="txManager" class="org.springframework.transaction.jta.JtaTransactionManager"/>
<!---->


<!--开启事务注解驱动，这样直接在类上使用@Transactional，注意public方法才有效，同一个上下文开启-->
<tx:annotation-driven transaction-manager="transactionManager"/>

  <!-- 启用aop自动代理，这样能够直接在java类使用aop注解定义切面 -->
  <aop:aspectj-autoproxy proxy-target-class="true" />


<!-- this is the service object that we want to make transactional -->
<!--要使用的切面pojo-->
<bean id="fooService" class="x.y.service.DefaultFooService"/>


<!--定义事务通知，引用事务管理器-->
<tx:advice id="txAdvice" transaction-manager="txManager">
<!-- the transactional semantics... -->
<tx:attributes>
<!-- all methods starting with 'get' are read-only -->
<tx:method name="get*" read-only="true"/>
<!-- other methods use the default transaction settings (see below) -->
<tx:method name="*"/>
</tx:attributes>
</tx:advice>

<!-- ensure that the above transactional advice runs for any execution
of an operation defined by the FooService interface -->
<!--定义切面-->
<aop:config>
<aop:pointcut id="fooServiceOperation" expression="execution(* x.y.service.FooService.*(..))"/>
<aop:advisor advice-ref="txAdvice" pointcut-ref="fooServiceOperation"/>
</aop:config>

小结：

当在spring上下文中配置开启事务注解驱动的时候，其实我们就可以直接在pojo上(一般在service层使用事务,所以为service层上的pojo)使用@Transactional注解开启该方法的事务功能。public方法才有效，否则无事务功能，即使开启注解

@Transactional这个注解其实可以允许有许多参数(更多参数请查阅官方文档)，比如配置事务传递性，什么异常会回滚(默认是发生unchecked异常时回滚，传递性为Required)。

事务在spring中是通过aop代理实现的。使用事务注解开启事务功能确实很方便，但是有些时候我们必须在spring配置文件中开启事务功能，比如，当这个事务功能应用范围比较大，复用性比较强，如果每次都使用注解驱动可能会比较冗余；或者这个类你没有源码，是第三方的类，这时使用xml结合spring的aop还是很方便的。

spring AOP

aop同样可以有两种方法来定义切面，一种使用注解，一种使用xml。

在spring文件中配置aop自动代理

<!-- 启用aop自动代理，这样能够直接在java类使用aop注解定义切面 -->
<aop:aspectj-autoproxy proxy-target-class="true" />

这时候就可以在pojo中使用注解来定义切面了。

xml配置方式则是在

开头定义切面。

被作为切面的pojo需要在配置文件中定义为bean

总结，我们可以看到无论是需要开启事务的方法还是定义切面的类，我们可以方向都是pojo，也就是无入侵式编程。
