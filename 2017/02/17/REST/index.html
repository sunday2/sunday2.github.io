理解REST

发表于 2017-02-17 | 分类于 翻译 |

原文

REST（Representation State Transfer:表述性状态转移）2000年在Roy Fielding的博士论文中被介绍和定义.REST是设计分布式系统的一种架构风格。 它不是一个标准，而是一组约束，例如是无状态的，具有客户端/服务器关系，以及统一的接口。 REST不是与HTTP严格相关，但它最常与它相关联。

REST的原则

资源(RESOURCES):暴露容易理解的目录结构URIS

表述性（Representations）：传输JSON或XML形式来表示数据对象和属性

消息(Messages)：明确使用HTTP方法(比如GET,POST,PUT,和DELETE)

无状态(Stateless):请求之间的交互没有在服务器存储客户端上下文。限制了状态依赖性和扩展性。客户端保存会话状态.

HTTP方法

使用HTTP方法将CRUD(create,retrieve,update,delete)操作映射到HTTP请求.

GET

检索信息.GET请求必须是安全和幂等的，意味着无论使用相同的参数多少次，结果是一样的(译者注：这里的用结果来表述可能有问题，http规范中get请求是说幂等，即使有副作用每次的副作用应该一样，因为如果每次请求的结果不一样，但是没有产生副作用也属于幂等)。它们可能有副作用，但是用户并不期待它们(译者注:这里副作用没搞清楚是啥),所以它们对于系统的操作的来说是不重要的。请求也可以是部分或者是有条件的。

检索ID为1的地址：
GET /addresses/1

POST

请求URI上的资源对提供的实体(entity)做一些操作(do something)。通常POST用来创建一个新实体，但它也可以用来更新实体.

创建一个新地址：
POST /addresses

PUT

将实体存储在URI。 PUT可以创建新实体或更新现有实体。 PUT请求是幂等的。 幂等性是PUT与POST请求之间的主要区别(译者注：PUT是幂等，POST不是幂等)。

修改ID为1的地址：

PUT /addresses/1

注意：PUT替换现有实体。 如果仅提供数据元素的子集，则剩余的将被空或空替换。

DELETE

请求删除资源; 然而，资源不必立即被移除。 它可以是异步或长期运行的请求。

删除ID为1的地址：

DELETE /addresses/1

HTTP状态代码

  • 1XX -infomational
  • 2XX -success
  • 3XX -redirection
  • 4XX -client error
  • 5XX -server error

媒体类型
Accept和Content-Type HTTP头可以用于描述在HTTP请求中发送或请求的内容。 如果客户端请求JSON中的响应，则客户端可以将接受设置为application / json。 相反，当发送数据时，将Content-Type设置为application / xml告诉服务端请求中发送的数据是XML。(译者注：原文为告诉客户端，觉得不妥)

译者注：

幂等性(Idempotence)：

通俗的讲就是一个资源无论请求多少次，其副作用都是一样的(如果没有副作用的话那么应该n次都没有副作用)。

幂等其实数学中的一个概念，维基百科上的定义：

在数学里，幂等有两种主要的定义。

二元运算：
设S为一具有作用于其自身的二元运算的集合，则S的元素s称为幂等的(相对于)当
s\s = s.
特别的是，任一单位元都是幂等的。若S的所有元素都是幂等的话，则其二元运算*被称做是幂等的。例如，并集和交集的运算便都是幂等的。

一元运算:
设f为一由X映射至X的一元运算，则f为幂等的，当对于所有在X内的x，
f(f(x)) = f(x).
特别的是，恒等函数一定是幂等的，且任一常数函数也都是幂等的。

上面四种请求中除了POST，其它都是幂等的。

副作用(side effect)

先总结一下，在GET,POST,PUT,DELETE四种请求中，只有GET没有副作用.
按译者自己的理解所谓的副作用就是有没有对资源造成改变，显然GET只是检索资源而没有对资源进行操作(只读),其它请求都有副作用，但是副作用并不是判断是否幂等的充分条件.如果相同的请求每次都有相同的副作用，那么该请求也是幂等的.注意这里强调相同的副作用.

POST和PUT区别

读者看了上面的文章可能感觉POST和PUT没有区别，都可以创建或更新已有的实体，但是其实有本质区别，POST是非幂等而PUT是幂等.POST请求的uri是接受实体的uri，使用该uri来创建或更新传递给服务端的实体。PUT请求的uri本身就是要创建或者更新的实体(所以客户端必须知道被处理的实体uri)。一个实际场景就是很多时候我们都是用数据库自增主键作为实体标志，这时候资源标识符(uri)是由服务端决定的，所以我们应该用POST，因为我们不知道要被处理的实体的id。

幂等不幂等有什么关系呢

我们很多时候因为网络等原因发送一个请求但是服务端并没有给予客户端反馈，这个时候我们可能会刷新一下浏览器再次发送方请求。那么问题来了，如果第一次请求其实服务端已经收到并且成功处理了该请求，只是在返回数据给客户端过程中因为网络原因延迟了，那么你就相当于发送了两次相同的请求。对于幂等请求来说发送多次相同的请求和发送一次请求是等价的，但是对于非幂等操作来说，就会产生一些意外的事故，特别是涉及金钱操作的时候，你原本只需要转100刀给别人，却转了两次，相当于转了200刀。所以有时候在chrome刷新页面的时候会提醒你是否再次发送方请求。
