spring之事务(1)

发表于 2017-06-13 | 分类于 java |

和数据据库打交道有一个重要概念就是事务，总结一下事务的一些知识点，强化记忆和理解。

事务概念

事务说到底就是用户定义的数据库操作序列，但是这组操作序列有个特点，就是要么全做要么全不做，是一个不可分割的工作单位。(是不是很容易理解)

还有重要一点，事务的开始和结束可以由用户显示控制，如果用户没有显示地定义事务，则DBMS会按缺省规定自动划分事务。实际上一般默认一条sql语句就是一个事务。

在网站开发中请求经过一系列业务逻辑判断表现为从数据库取数。

事务格式

BEGIN TRANSACTION
COMMIT
ROLLBACK

BEGIN TRANSACTION事务开始的标志，事务结束标志要么是COMMIT要么ROLLBACK。决定是COMMIT还是ROLLBACK在于事务执行过程有没有发生某种故障。没有发生故障则将对数据库的更新写回到磁盘的物理数据库上，否则ROLLBACK，回到一开始的状态(撤销已经执行的操作)。

事务的特点

ACID很出名，就不多说。主要说一下ISOLATION隔离性，隔离性在spring的事务管理中体现在事务传递上。隔离性表示两个事务不互相影响，不互相干扰。

事务是恢复和并发控制的基本单位。所以与事务息息相关的就是恢复和并发控制

数据库故障类型

事务内部故障

有的可以通过程序本身发现，有的是非预期的。
例子：转账而余额不足

解决办法：事务撤销(UNDO)

系统故障(软故障)

系统故障指造成系统停止运转的任何事件，使得系统要重新启动。
例子：硬件错误(CPU故障)，操作系统故障，DBMA代码错误，系统断电等。

影响正在执行的所有事务，但不破坏数据库主要造成主存，特别是缓冲区(内存)的内容丢失，所有运行事务非正常终止。

数据库缓冲区为内存。

解决办法:
(1)所有非正常终止的事务(未提交的)回滚，强行撤销UNDO所有未完成的事务。
(2)有一部分已经完成的事务一部分或者全部留在缓冲区，未回写到磁盘上的物理数据库中，使对数据库的修改部分或全部丢失，需要REDO重做所有已提交的事务。(这里猜测是一般数据库先写日志再提交)

介质故障(硬故障，磁盘故障)

硬故障指外存故障。

例子：磁盘损坏，磁头碰撞，瞬时强磁场干扰等。

与软故障一个重要区别是硬故障会损坏物理数据库。

计算机病毒

总结

总结总共就是两大类：损坏物理数据库和没损坏物理数据库。

恢复数据库概括： 冗余(通过存储在系统别处的冗余数据来重建，其实就是备份之类的)

建立冗余数据的方法

  • 数据转储
  • 登记日志文件

数据转储

  • 静态转储：系统中无运行事务时进行的转储操作。弊端：降低数据库的可用性。

  • 动态转储：转储期间允许对数据库进行存取和修改。转储和用户事务可以并发执行。弊端：转储结束后不能保证数据正确有效，具有延时性。

登记日志文件

  • 以记录为单位的日志文件
  • 以数据块为单位的日志文件
记录为单位的日志文件

日志文件登记内容：

  • 各个事务的开始标记
  • 各个事务的结束标记
  • 各个事务的所有更新操作

每条日志记录内容：

  • 事务标识(标明哪个事务)
  • 操作的类型(插入，删除，修改)
  • 操作对象(记录内部标识)
  • 更新前数据的旧值(对于插入操作来说为空)
  • 更新后数据的新值(对于删除操作来说为空)
以数据块为单位的日志文件

内容包括事务标识和被更新的数据块。

日志文件作用
  • 事务故障恢复和系统故障恢复必须使用日志文件
  • 动态转储必须建立日志文件，因为动态转储具有延时性，动态转储文件和日志文件结合才能有效恢复数据库。
  • 静态转储也可以建立日志文件。
登记日志文件的两条原则
  • 登记的次序必须严格按照并发事务执行的时间次序
  • 必须先写日志文件，再写数据库(万一在这两个操作中间发生故障才不至于数据丢失)
